📍 L1 范围声明

本子系统只做 Amazon Brand Store 引流与转化的多维分析、Tag 与联盟的全生命周期管理；
不做广告投放执行（归 amazon-ads-system）、不做订单履约（归 amazon-sales-system）、不做品牌注册与 Listing 发布。

1
设计原则（4 条）
围绕旗舰店子系统边界与目标，定义能力层的统一设计原则
01
聚焦"引流 → 转化 → 复盘"主航道
所有能力必须能落到这一主航道的某一段；偏离主航道的能力（如 Listing 发布、订单履约）一律归外部子系统。
02
L1 不出现菜单粒度
能力到二级即可（共 14 项）；菜单与角色权限矩阵在 flagship-store_menu_v1.md 的 L2 解决。
03
跨域数据明示边界
与 amazon-sales / amazon-ads / off-site-ads / OC 的接口在能力图中以"虚线"显式标注，禁止默认依赖。
04
能力 = 业务方的语言
统一使用业务语义术语，避免 PM 与研发侧各自定义概念造成沟通偏差。

4 大能力域详解（含价值描述 / 关键支撑 / MVP）
每域含 4–5 项二级能力 + 业务定位 + 关键决策
① 引流归因 Inbound & Attribution 主用户 · CP（渠道推广）
业务定位：把"流量怎么进来 / 怎么追踪 / 怎么算钱"这件事，从 Excel 化人工搬到系统化自动。
二级能力 价值描述 关键支撑 MVP
C1.1Tag 自助创建 CP 在 ERP 内一站式创建 Amazon Attribution Tag/UTM，按 品牌-站点-SKU-渠道-日期 规范自动生成 命名校验器 + Attribution API（待评估） P0
C1.2联盟管理 多平台 CSV/Excel 导入 → ASIN→SKU 映射 → 自动算佣金（Sales × 比率）/ ROAS / 预算进度 导入器插件化、ASIN-SKU 主数据 P0
C1.3Live 排期与回流 直播档期登记、亚马逊官方直播红人记录、播后数据回流 当前手工，P1 接入 SP-API P1
C1.4命名规范校验与治理 历史脏 Tag 治理、新建强约束、违规告警 命名规范字典（系统资产） P0
📌 关键决策：联盟属于"站外 Other"，Live 属于"站内"，Tag 属于"站外 Other"。这一分类直接决定看板（C3.3）的渠道分桶。
② 旗舰店运营 Store Ops 主用户 · SO（旗舰店运营）
业务定位：维护"站点 → 场景 → 页面 → SKU"这棵 4 层的业务树，所有看板的下钻路径都依赖它。
二级能力 价值描述 关键支撑 MVP
C2.1站点-场景维护 7 站点（北美 2 + 欧洲 5）× 6 场景（公共/居家/户外/观影/新潮/季节）+ 可扩展（活动场景） 场景类型字典（基础/活动） P0
C2.2页面-场景映射 每个 Brand Store 页面归属唯一场景，新建 / 变更 / 历史快照 映射版本快照表（按月固化） P0
C2.3重点品 Hero SKU 标记 SO 手动标记某 SKU 为"重点品"，看板自动汇总 SKU 维度元数据 P0
C2.4目标管理 月度 / 季度 / 年度目标（Sales 60% + CTR 20% + ADT 15% + UPT 5%）维护与达成跟踪 目标表（站点 × 维度 × 周期） P0
③ 数据分析 Insight 主用户 · DA / BO（数据分析师 / 业务负责人）
业务定位：替代当前"Excel + DeepSeek 跑代码"的手工流，提供 7 大看板的实时化、统一口径化。
二级能力 价值描述 关键支撑 MVP
C3.1站点总览看板 单站点 / 北美聚合 / 欧洲聚合 / 全站聚合；周/月时间维度；环比 + 同比 时间筛选器 + 聚合服务 P0
C3.2场景与页面看板 场景维度月度对比、场景下钻到页面、页面下钻到流量组成 C2.1 / C2.2 映射 P0

4
统一指标计算服务（贯穿 C3.x 所有看板）
所有比率类指标必须由后端计算，前端只展示，杜绝 Excel 时代的口径漂移
指标 公式 聚合方式
Sales / Orders / Units / Visitors / Views sum() 求和
CVR · 转化率 Orders / Visitors sum/sum 重算
ASP · 客单价 Sales / Orders sum/sum 重算
UPT · 件单 Units / Orders sum/sum 重算
NTS 占比 · 新客占比 NTS / Visitors sum/sum 重算
Daily Views / Daily Sales · 日均 Total / 周期天数 平均值
ADT · 平均停留时长 avg(dwell time) 平均值
⚠️ 关键决策：所有 sum/sum 重算 类指标禁止在前端再次计算或对率值做平均；这是替代"Excel + DeepSeek 手工流"的根本能力，写入 K3 关键约束。

💡 v0.1 能力架构的三大设计意图
1. 主航道收敛 = 拒绝功能膨胀

4 大能力域全部围绕"引流→转化→复盘"主航道。Listing 发布、订单履约、广告投放执行等被显式排除，避免子系统变成"亚马逊万能后台"。

2. 业务术语 ≠ 技术术语

14 项二级能力直接复用业务方 Excel 中的称谓（如 Hero SKU、场景、流量组成）；技术名词（指标计算服务、维度模型、映射快照）只在关键支撑列出现，不污染业务视角。

3. MVP 即 P0 13 项，可 8 周内交付

P0 覆盖业务方最痛的"自助 Tag + 联盟导入 + 7 大看板（不含跨店占比） + 命名规范"；P1 用于解决依赖外部数据 / 待评估的项（直播、跨店占比、报表自动化）。
